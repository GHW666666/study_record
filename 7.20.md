【项目背景】
依托微信小程序生态搭建流量承接体系，围绕共享相册能力打造「夏日足迹
地图」裂变活动：用户创建共享相册组队，上传照片经策略识别点亮地标、
积分排行，最终生成回忆视频，以社交分享驱动裂变增长、以会员能力承接
商业化转化。活动同时投放在网盘 APP 端内与网盘微信小程序，需处理双端
交互与链路差异。

【我的职责与产出】（可按实际情况删改）
1. 负责活动前端小程序侧的技术预研与方案落地，主导完成小程序 web-view
   与 H5 混合开发的完整链路验证，为活动 H5 页面嵌入小程序提供可行性
   结论与标准接入方案。

2. 攻克 web-view 与 H5 双向通信难题：
   - 验证并确认 wx.miniProgram.postMessage 的消息投递时序特性（仅在
     后退/组件销毁/分享时机批量投递，非实时），发现「转发时
     onShareAppMessage 同步返回、无法读到本次投递最新数据」的时序
     风险，并给出规避方案；
   - 设计「原生页 → getCurrentPages 回传 → web-view src 拼 hash 重载
     → H5 监听 location.hash」的小程序向 H5 实时传参方案，解决
     navigateBack 无法携参、URL query 传参会触发整页刷新的问题。

3. 打通登录态透传链路：实现小程序内 BDUSS/STOKEN → access_token 的
   换取流程（带 MD5 签名的 toaccesstoken 接口），使嵌入的 H5 活动页
   免登录获取用户身份；同时识别出凭证明文拼 URL 的安全风险，推动
   正式方案改用服务端 pantoken 换取方式。

4. 完成微信邀请分享回流链路：H5 触发邀请参数 → 小程序 bindmessage
   接收 → onShareAppMessage 拼装分享卡片（小程序活动框），支持被邀请
   者点击直达活动页并携带 inviteId 完成组队归因。

5. 验证小程序 web-view 内 H5 调端能力（唤起网盘 APP / scheme 跳转）
   的可用边界，为「端内外双端推广、链路互跳」提供技术依据。

6. 遵循小程序分包架构，活动模块独立分包接入，控制主包体积。

三、技术亮点清单（面试展开讲，按 STAR 准备）
──────────────────────────────

★ 亮点 1：postMessage 时序风险的发现与验证
  S：活动需要 H5 页把邀请参数传给小程序用于分享卡片
  T：验证 web-view postMessage 能否在分享时拿到最新数据
  A：写 demo 实测发现微信只在「后退/销毁/分享/复制链接」时机批量投
     递消息，而 onShareAppMessage 是同步返回，setData 来不及生效，
     分享时读到的是上一次的数据
  R：提前暴露风险，改为提前投递参数/兜底默认分享文案的方案，避免
     线上分享参数丢失

★ 亮点 2：小程序原生页 → H5 的实时传参方案
  问题：navigateBack 无法带参；改 query 会导致 web-view 整页刷新
  方案：getCurrentPages 拿上一页实例回调传参，src 拼
        #params=xxx&t=时间戳 触发 web-view 重载，H5 读 location.hash
        即可拿到参数且不刷新页面主体
  关键词：getCurrentPages、hash 通信、web-view src 变更机制

★ 亮点 3：登录态透传与安全治理
  问题：嵌入的 H5 活动页需要网盘登录态
  方案：小程序端取 BDUSS/STOKEN，调 toaccesstoken 接口（MD5 签名）
        换取 access_token 拼入 H5 地址
  安全意识加分点：识别出 BDUSS 明文放 URL 的凭证泄漏风险，推动正式
        方案改为服务端 pantoken 换取、URL 不落敏感凭证

★ 亮点 4：三类分享回流链路设计（可讲业务复杂度）
  - 活动首页分享：微信内分享小程序活动框、其余渠道分享 H5 链接
  - 组队邀请分享：复用共享相册邀请链路，被邀请者进入需处理
    已加入其他小队/小队满员/已是成员/活动已结束 4 种边界
  - 弹窗优先级管理：确认加入 > 榜单公示 > 成员变动 > 积分变动

★ 亮点 5：双端适配
  同一活动在网盘 APP 端内（H5 + 调端）与微信小程序（web-view 嵌 H5）
  两套宿主环境运行，上传面板、分享面板、会员收银台等链路双端差异适配

四、可能被追问的问题（提前准备）
──────────────────────────────

1. 为什么用 hash 而不是 query 传参给 H5？
   → query 变化会导致 web-view 认为是新页面整体重新加载；hash 变化
     配合时间戳既能触发 src 更新又能让 H5 用 onhashchange/location.hash
     轻量读取，体验更好。

2. postMessage 为什么不实时？怎么解决的？
   → 微信官方设计如此（防滥用），消息在特定生命周期时机批量投递；
     解决思路：H5 尽早 postMessage（不要等点分享才发）、小程序端
     onShareAppMessage 里做兜底参数。

3. BDUSS 直接拼 URL 有什么问题？
   → 登录凭证暴露在链接里，可能被日志、代理、分享行为泄漏；正式方案
     应由服务端换取一次性 token（pantoken）再下发带鉴权的 URL。

4. 小程序分包为什么这么做？
   → 活动是非首屏低频页面，独立分包按需加载，避免撑大主包影响
     启动性能（主包有 2M 限制）。
